  answer (constraintВ 3): hardware identity sidesteps IP entirely.
  - The honest ceiling: the same search surfaced commercial "TPM bypass 2026" spoofers. A determined attacker with new physical hardware or
  a hardened VM will get through device signals. The robust backstop for those is server-side behavioural/account-graph correlation, which
  the attacker can't touch at all.

     Plan вҖ” Device Identity & Ban-Evasion Detection (No-TPM, Open-Source-Safe)

     в–Һ Status: theoretical design only. No code to be written yet. This document is the plan.

     Context

     Players who are suspended for misbehaviour try to evade by making new accounts and/or
     disguising their machine and connection. We want a system that reliably re-identifies the
     same person/device behind a new account so suspensions can actually be enforced.

     Hard constraints, from the requirements discussion:

     1. Non-intrusive вҖ” no kernel driver, no boot-time component, no forced hardware gate.
     2. Open-source-safe вҖ” the system must keep working even with all client code public
     (Kerckhoffs's principle: security must not depend on the code being secret).
     3. VPN-tolerant вҖ” many legitimate users are behind VPNs/CGNAT/shared IPs.
     4. VM-aware вҖ” some evaders spoof a "fresh machine" inside a virtual machine.
     5. MITM-resistant вҖ” some evaders tamper with the fingerprint in flight.

     Decisions already made:
     - No TPM / no hardware root of trust (don't want to depend on or require it).
     - Cross-platform target (Windows + Linux; macOS droppable вҖ” small population).
     - Enforcement leaning towards prevent login on detection (treated as downstream; see below).
     - Collection must be disclosed to users, never silent. Retention form TBD.

     The current client already sends basic machine info in plain text over unencrypted HTTP,
     Windows-only вҖ” unusable for this purpose. That channel is the first thing to replace.

     ---
     The central design truth (read this first)

     Without a hardware root of trust, you cannot cryptographically prove that a fingerprint
     reflects the real machine. The client runs on hardware the attacker owns; with open-source
     code they can read exactly what is collected and run a modified client that reports anything.
     No amount of TLS, nonces, or signing changes this вҖ” those protect the channel, not the
     honesty of the source.

     Therefore the system is probabilistic, not cryptographic, and is built on two principles:

     - Server-authoritative. The client only collects and reports. It never decides anything
     and never enforces anything. NEXUS is the sole trust boundary and the sole decision-maker.
     - Untrusted inputs вҶ’ correlation engine. Every client-reported signal is treated as an
     untrusted hint feeding a weighted similarity/risk score, cross-checked against signals the
     client cannot author (network, TLS, behaviour). We never treat a client value as proof.

     The realistic goal is to defeat the ~95% of casual evaders cheaply and invisibly, and make
     evasion expensive for the rest вҖ” not to be unbeatable. (See "Honest limitations" below.)

     Four layers, in priority order of build + robustness:

     - Layer 1 вҖ” Hardened, open-source-safe transport (foundation; replaces plaintext HTTP).
     - Layer 2 вҖ” Composite hardware fingerprint (the client collector; cost-raiser).
     - Layer 3 вҖ” Server-side signals the client can't forge (the robust backbone).
     - Layer 4 вҖ” VM & spoof detection (risk multipliers over Layers 2вҖ“3).

     A note on emphasis: because Layer 2 is forgeable, Layer 3 is where durable detection lives.
     Layer 2 raises the cost of casual evasion and catches the lazy; Layer 3 catches the determined
     because it draws on data the attacker does not control. Build Layer 1 first, then Layer 3 and
     Layer 2 in parallel, then Layer 4.

     ---
     Layer 1 вҖ” Hardened, open-source-safe transport

     This is what makes the "open-source + nonce" idea actually work, and it directly answers
     constraint 5 (MITM) and the network half of spoofing. All of it is safe to publish.

     - TLS 1.3 + certificate / public-key pinning. WILLOWMAKER pins NEXUS's public key so a
     transparent proxy cannot sit in the middle and re-sign traffic. This alone is the single
     biggest upgrade over the current unencrypted HTTP and defeats passive sniffing and most
     active network MITM.
     - ChallengeвҖ“response nonce. Client calls GET /attest/challenge вҶ’ NEXUS returns a random
     nonce with a short TTL, bound server-side to the authenticated session. The client must echo
     this nonce inside the signed payload. NEXUS rejects unknown / expired / already-used nonces.
     This defeats capture-and-replay of a previously valid fingerprint.
     - Session-bound signature. After normal authenticated login, the payload
     {fingerprint components + nonce + timestamp} is signed with the session credential (HMAC
     with a per-session key issued at login, or the session token). This authenticates the report
     as coming from this logged-in session, so an outsider cannot inject a fingerprint against
     someone else's account.

     What this buys, precisely (be honest in the design docs):
     - вң… Stops outsiders: passive sniffers, network MITM, replay, cross-account injection.
     - вқҢ Does not stop the account owner running a modified client from submitting fabricated
     components. That residual risk is handled probabilistically by Layer 3 cross-checks, never
     by the crypto. State this boundary explicitly wherever the protocol is documented.

     This is exactly the "nonce from NEXUS used so it doesn't matter that the code is open" idea вҖ”
     correct, and it composes cleanly. The only thing it can't do alone is guarantee source honesty,
     which is why the no-TPM choice pushes weight onto Layer 3.

     and send the set, so the server can compute partial similarity ("7 of 9 stable components
     match a suspended device вҶ’ 0.9").

     Per-platform sources:
     - Windows: SMBIOS via WMI/GetSystemFirmwareTable, disk serials via IOCTL/WMI, registry
     MachineGuid, EDID from registry, physical NIC enumeration (filter virtual adapters).
     - Linux: /sys/class/dmi/id/* (product/board UUID + serials), /etc/machine-id, udev disk
     serials, /proc/cpuinfo, sysfs NICs.
     - macOS (if kept): IOPlatformUUID, hardware serial, model, primary MAC.

     Hashing / privacy hygiene:
     - Hash each component with a single fixed application-wide salt (pepper), not a per-user
     salt вҖ” per-user salting would make identical hardware hash differently and break matching.
     - Note honestly that low-entropy serials are brute-forceable from a hash, so client-side hashing
     is privacy hygiene, not a security control. If stronger privacy is wanted, the server can
     HMAC with a server-held pepper (client sends raw over the pinned TLS channel) вҖ” a trade-off to
     decide later (ties into the retention question you flagged).

     Where it runs: collection logic lives in WILLOWMAKER (the launcher, already cross-platform
     and the natural place for machine inspection); hon_client need not change beyond passing a
     session/nonce through if convenient. NEXUS gains the ingest + storage endpoints.

     ---
     Layer 3 вҖ” Server-side signals the client can't forge (the backbone)

     These are observed by NEXUS, not reported by the client, so they are immune to client
     modification, VM spoofing, and in-flight tampering вҖ” and largely to VPNs. This is where durable
     re-identification comes from.

     - Network intelligence. Source IP, ASN, geolocation, and VPN/proxy/datacentre detection
     (commercial feed e.g. IPQS/MaxMind, or open lists). VPNs do not defeat this вҖ” a VPN exit IP
     is itself a flaggable property, and datacentre ASNs are a strong signal. Correlate accounts
     seen from the same residential IP / subnet. Weak alone (CGNAT, families), valuable in aggregate.
     - Transport fingerprinting (JA3/JA4, HTTP/2 settings). The TLS handshake and HTTP client
     stack produce a fingerprint the evader usually doesn't think to vary вҖ” a free passive signal.
     - Behavioural & account-graph correlation (highest long-term value, hardest to evade):
     social graph (who they queue/party with), play schedule + timezone, hero/champion and build
     preferences, chat style, matchmaking patterns, registration metadata (email/phone, disposable-
     email detection), and account-creation velocity per IP/subnet. A returning banned player tends
     to rejoin the same friends and habits вҖ” none of which the client controls.
     - Registration friction. Email/phone verification raises account-farming cost directly.

     ---
     Layer 4 вҖ” VM & spoof detection (risk multipliers)

     Without TPM these are heuristic and individually beatable, but each raises cost and together
     form a useful risk multiplier вҖ” never a standalone ban.

     - Hypervisor CPUID leaf / vendor strings; SMBIOS vendor = VMware / VirtualBox / QEMU /
     Microsoft Corporation Virtual Machine.
     - VM MAC OUIs (VMware 00:05:69/00:0C:29/00:50:56, VirtualBox 08:00:27, Hyper-V 00:15:5D).
     - VM-specific devices/drivers, absence of physical-display EDID, timing anomalies.
     - Inconsistency detection (cheap and powerful): client claims contradict server-observed
     reality вҖ” e.g. claimed timezone/locale vs IP geolocation, claimed OS vs TLS fingerprint,
     "physical" NIC with a VM OUI. Mismatches are strong tamper indicators even when each field
     looks individually plausible.

     Treat "detected VM" or "internal inconsistency" as a multiplier on the match/risk score,
     and a clean brand-new fingerprint emerging from a detected VM as inherently high-risk.

     ---
     The correlation engine (NEXUS)

     The core deliverable. On each login/registration, build a record and score it against all known
     device/identity records:

     1. Per-device record: the set of hashed components + observed network/TLS signals + VM/
     inconsistency flags, timestamped, linked to the account(s) seen using it.
     2. Identity graph: accounts вҶ” devices вҶ” network clusters вҶ” behavioural clusters, edges
     weighted by signal strength. A suspension attaches to an identity cluster, not just one
     account.
     3. Weighted fuzzy match: similarity = weighted sum of component matches (LayerВ 2) + network/
     behavioural proximity (LayerВ 3), scaled by Layer-4 multipliers. Produce a confidence score
     and the contributing evidence.
     4. Threshold вҶ’ action (policy-tunable): below T1 ignore; T1вҖ“T2 soft/flag; above T2 act.
     The threshold is the only place enforcement posture (below) actually couples back into the
     fingerprinting design вҖ” it sets how confident the engine must be before acting.

     Keep thresholds and weights server-side and unpublished even if the protocol is open: that
     is not security-by-obscurity of the mechanism, just not handing evaders your exact tuning.

     ---
     Enforcement & privacy (downstream вҖ” kept deliberately short)

     These sit on top of the engine and don't change its core design; only the threshold couples in.

     - Enforcement: "prevent login on detection" is supportable вҖ” gate login on the cluster's
     suspension state when confidence вүҘ T2. Recommendation: pair it with a confidence threshold and
     an appeal path, because shared machines (families, cyber-cafГ©s, dorms) are the main source
     of false positives in any device-correlation system. Graduated responses (re-verification,
     restricted matchmaking) are a softer option to keep in reserve.
     - Privacy: disclose collection in EULA/privacy policy (never silent, per your call).
     Document a lawful basis (anti-abuse legitimate interest) and a retention period. Retention
     form (salted hashes vs HMAC-with-server-pepper vs richer raw signals) trades privacy against
     correlation power and can be decided when we build Layer 2/3 вҖ” flagged, not blocking.

     ---
     Honest limitations (the ceiling)

     State these plainly so expectations are calibrated:

     - A determined evader on new physical hardware is effectively a new person to Layer 2 вҖ” only
     Layer 3 (behaviour/social graph/network) can re-link them.
     - A hardened VM that spoofs SMBIOS/MACs/timezone consistently can defeat Layers 2 andВ 4.
     - The open-source client can be modified to report anything; commercial "spoofers" exist even
     against TPM-backed systems. Crypto (LayerВ 1) cannot fix source honesty without hardware trust.
     - Conclusion: device fingerprinting raises cost and catches the casual majority; durable
     enforcement against the determined minority rests on server-side behavioural correlation.

     (If the no-TPM stance ever softens, TPM attestation slots in as an opportunistic Layer-2+
      signal вҖ” "real TPM present and consistent" becomes a high-trust booster вҖ” without changing any
      other layer. Documented here as a future option only.)

     ---
     Phased rollout

     1. Layer 1 first вҖ” replace plaintext HTTP with pinned TLS + nonce challenge-response +
     session-bound signature. Foundational; everything else rides on it.
     2. Layer 3 backbone вҖ” server-side network/TLS/behavioural ingest + identity graph. Highest
     robustness, no client changes beyond what Layer 1 provides.
     3. Layer 2 collector вҖ” cross-platform component collection in WILLOWMAKER + ingest in NEXUS.
     4. Layer 4 + correlation engine вҖ” VM/inconsistency flags and the weighted matcher.
     5. Enforcement + privacy policy вҖ” thresholds, login gating, appeals, disclosure, retention.

     ---
     Verification (when built)

     - Transport: attempt MITM with a proxy (mitmproxy) against a pinned build вҶ’ connection must
     fail; replay a captured payload вҶ’ rejected on nonce; tamper a field in flight вҶ’ signature fail.
     - Collector: run on Windows + Linux (+ macOS if kept), confirm stable components are stable
     across reboots and that changing one component drops similarity by the expected weighted amount.
     - VM detection: run inside VMware/VirtualBox/Hyper-V/QEMU вҶ’ VM flags + inconsistency flags fire.
     - Correlation: seed a suspended identity; simulate evasion attempts (new account same machine;
     new account + spoofed MAC; new account in a VM; new account + VPN) and confirm each crosses or
     stays under the intended threshold, with no false positive for a genuinely distinct second user
     on a shared machine.
